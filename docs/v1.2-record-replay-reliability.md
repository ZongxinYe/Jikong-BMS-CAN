# v1.2：记录完整性与回放末端保持

## 修复内容

v1.2 修复回放完成事件早于 GUI 帧队列消费的问题。控制器收到完成、停止或错误终态后，会等待已经入队的帧全部进入解码管线，再切换为空闲状态。回放自然结束或用户停止数据源时保留最后快照、BMS 地址、波形选择和曲线；点击“清空”或启动新数据源时才重置。

SQLite 继续使用 schema v2 `raw_only` 格式，不增加派生信号表。正常停止依次执行：停止接收新记录项、排空后台队列、提交 `sessions.ended_at` 和地址集合、执行 `PRAGMA wal_checkpoint(TRUNCATE)`。

## 记录状态

- `记录 #N`：会话正在写入。
- `已保存 #N`：队列、会话元数据和 WAL checkpoint 均已完成，主 `.sqlite3` 可独立复制。
- `已停止 #N（需保留 WAL）`：已停止，但 checkpoint 受读锁或 SQLite 错误影响；主库、同名 `-wal`、同名 `-shm` 必须一起保存。
- `记录异常停止 #N`：记录器发生队列满、SQLite writer error 或停止超时；工具提示保留路径、错误和已写帧数。

独立诊断日志默认位于：

```text
%LOCALAPPDATA%\BMS CAN Monitor\logs\recording-audit.jsonl
```

日志记录 started、stopped、shutdown、failed 和错误后收尾结果。日志不可写不会中断 CAN 只读监测或 SQLite 原始帧记录。

## 未收尾数据库

打开 `sessions.ended_at IS NULL` 的会话时，程序会提示该文件可能仍在记录、程序曾异常退出，或复制时遗漏了 WAL/SHM。用户确认后可以只读回放当前可见帧，但程序不会把它标记为完整记录。

异常退出后恢复数据时，应保持以下三个文件同名并位于同一目录：

```text
record.sqlite3
record.sqlite3-wal
record.sqlite3-shm
```

不要手工删除 WAL 或只重命名其中一个文件。
