# Phase 5：控制帧与安全保护

## 协议依据

控制命令依据 `极空BMS-Can协议V2.1.pdf` 第 34-35 页的 `Ctrl_INFO` 定义：

- 扩展帧 ID：`0x18F0F428`。
- Byte 0 `MaskCode`：bit0 充电控制、bit1 放电控制、bit2 均衡控制。
- Byte 1 `ChgSw`：`0` 关闭，`1` 开启。
- Byte 2 `DchgSw`：`0` 关闭，`1` 开启。
- Byte 3 `BanlanSw`：`0` 关闭，`1` 开启。DBC 和代码统一使用修正后的名称 `BalanSw`/`balance_on`。
- Byte 4-7 为协议示例中的 `XX` 保留字节。本程序固定发送 `00 00 00 00`，避免发送未初始化数据。

PDF 示例 `05 01 01 01 XX XX XX XX` 在本程序中编码为：

```text
ID   0x18F0F428（扩展帧）
DATA 05 01 01 01 00 00 00 00
```

`MaskCode=0x05` 表示充电和均衡目标状态生效，放电字节即使为 `01` 也不生效。

## 发送保护

控制发送必须同时满足以下条件：

1. 数据源为真实 CANalyst-II，且通道已经启动。
2. CAN 通道使用正常模式；只听模式和自测模式禁止发送。
3. 当前配置为默认 BMS 设备地址 `0`。
4. 最近 3 秒内成功解码过目标 BMS 报文。
5. 至少选择一个 MaskCode 控制项。
6. 控制工作区已经经过首次风险确认解锁。
7. 本次命令已经经过包含 ID、DATA 和目标状态的二次确认。
8. 一次性授权令牌未过期、未使用，且命令内容与确认时完全一致。
9. “发送请求”审计记录已经成功写入磁盘。

任一条件不满足时，`GuiController` 在调用 `ControlCAN.dll` 前拒绝命令。GUI 按钮是否可用不是唯一保护；底层安全门独立执行同样的校验。

## 默认地址限制

PDF 只明确给出了默认控制 ID `0x18F0F428`，没有明确说明非零 BMS 设备地址时，外设到 BMS 的目标地址字段如何变化。该报文方向与 BMS 主动上报方向相反，不能直接套用“完整 ID 加设备地址”。

因此 Phase 5 只允许设备地址 `0`。地址 `1..11` 的监测解码仍可使用，但控制页显示不可发送。后续必须通过厂家说明或真机抓包确认下行寻址规则，再单独解锁并增加测试。

## 双重确认

- 工具栏“控制锁定”默认禁用。只有真实 CAN、正常模式、默认地址且收到近期 BMS 报文后才可点击。
- 第一次确认只解锁控制工作区，不产生 CAN 帧。
- 每次单击“确认并发送”都会显示本次扩展帧 ID、8 字节 DATA 和实际生效项。
- 用户确认后生成一个 30 秒有效、只能使用一次、与当前命令指纹绑定的随机令牌。
- 任何 Mask 或目标状态变化都会使原令牌失效。
- Phase 5 不提供控制帧周期发送，单次确认只发送一帧。

## 审计记录

控制审计默认写入：

```text
logs/control-audit.jsonl
```

每行是一个 UTF-8 JSON 对象，阶段包括：

- `authorized`：二次确认已经完成。
- `send_requested`：发送前审计已落盘，即将调用适配器。
- `succeeded`：适配器返回成功。
- `failed`：适配器发送失败或返回异常状态。
- `rejected`：连接、模式、地址、近期报文、Mask 或授权校验失败。

同一批事件还会显示在“运行事件”页；SQLite 会话记录开启时，也会写入 `events` 表。即使 SQLite 记录未开启，独立 JSONL 审计仍会保留。

## 测试

```powershell
$env:QT_QPA_PLATFORM = "offscreen"
python -B -m pytest -p no:cacheprovider
```

自动化测试覆盖 PDF 示例编码、保留位、非法 Mask、默认地址限制、令牌过期/复用/命令篡改、未连接、只听模式、过期 BMS 状态、成功发送事件链、审计日志和默认锁定的控制 UI。

自动化测试使用假适配器，不会连接或控制真实 BMS。首次真机发送应在台架环境中进行，并先确认充放电回路、接触器和负载处于可控状态。
